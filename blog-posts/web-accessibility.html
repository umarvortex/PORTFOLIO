Web Accessibility Best Practices

Web accessibility is the practice of ensuring websites, tools, and technologies are designed and developed so that people with disabilities can use them effectively. Beyond being a legal requirement in many countries, creating accessible websites is simply the right thing to do—expanding your reach to the over one billion people worldwide with disabilities.

This guide outlines essential guidelines and best practices for building inclusive and accessible websites that everyone can use, regardless of their abilities or the tools they use to navigate the web.

Why Accessibility Matters

Inclusivity

Everyone deserves equal access to digital content and services

Legal Compliance

Many regions require accessibility compliance (ADA, AODA, EAA)

Business Impact

Reach more users and improve SEO and user experience

Understanding WCAG

The Web Content Accessibility Guidelines (WCAG) are the industry standard for accessibility. Developed by the W3C's Web Accessibility Initiative (WAI), these guidelines help make web content more accessible to people with disabilities. The guidelines are organized around four key principles:

Perceivable

Information and user interface components must be presentable to users in ways they can perceive. This means providing text alternatives for non-text content and creating content that can be presented in different ways.

Operable

User interface components and navigation must be operable. This means making all functionality available from a keyboard and giving users enough time to read and use content.

Understandable

Information and the operation of the user interface must be understandable. This means making text readable and predictable, and helping users avoid and correct mistakes.

Robust

Content must be robust enough to be interpreted reliably by a wide variety of user agents, including assistive technologies. This means maximizing compatibility with current and future tools.

Key Accessibility Best Practices

1. Semantic HTML

Using the right HTML elements for their intended purpose ensures that content is properly structured and meaningful to both browsers and assistive technologies.

Instead of this:

<div class="header">
  <div class="nav-item">Home</div>
  <div class="nav-item">About</div>
  <div class="nav-item">Contact</div>
</div>

<div class="title">Welcome to Our Website</div>
<div class="content">
  This is some important information.
</div>

<div class="button" onclick="submitForm()">Submit</div>

Do this:

<header>
  <nav>
    <a href="/">Home</a>
    <a href="/about">About</a>
    <a href="/contact">Contact</a>
  </nav>
</header>

<h1>Welcome to Our Website</h1>
<main>
  <p>This is some important information.</p>
</main>

<button type="submit">Submit</button>

2. Keyboard Accessibility

All interactive elements must be accessible using only a keyboard, as many users cannot use a mouse. This includes ensuring proper focus states and logical tab order. 

/* Make focus states visible and consistent */
:focus {
  outline: 2px solid #4a90e2;
  outline-offset: 2px;
}

/* Custom component focus styles */
.custom-button:focus {
  outline: 2px solid #4a90e2;
  box-shadow: 0 0 0 4px rgba(74, 144, 226, 0.3);
  border-color: #4a90e2;
}

Always ensure that custom components are keyboard accessible: 

// Example: Making a custom dropdown keyboard accessible
const Dropdown = () => {
  const [isOpen, setIsOpen] = useState(false);
  const [selectedIndex, setSelectedIndex] = useState(0);
  const options = ['Option 1', 'Option 2', 'Option 3'];
  
  const handleKeyDown = (e) => {
    switch(e.key) {
      case 'ArrowDown':
        e.preventDefault();
        setSelectedIndex((prev) => Math.min(prev + 1, options.length - 1));
        break;
      case 'ArrowUp':
        e.preventDefault();
        setSelectedIndex((prev) => Math.max(prev - 1, 0));
        break;
      case 'Enter':
      case 'Space':
        e.preventDefault();
        if (isOpen) {
          // Select the option and close dropdown
          setIsOpen(false);
        } else {
          setIsOpen(true);
        }
        break;
      case 'Escape':
        e.preventDefault();
        setIsOpen(false);
        break;
    }
  };
  
  return (
    <div 
      role="combobox" 
      aria-expanded={isOpen}
      aria-haspopup="listbox"
      tabIndex={0}
      onKeyDown={handleKeyDown}
    >
      {/* Dropdown implementation */}
    </div>
  );
};

Testing Tip

Try navigating your website using only the keyboard. Can you access all interactive elements? Is the focus visible at all times? Can you easily understand where you are on the page?

3. Images and Alternative Text

Always provide descriptive alternative text for images to make them accessible to screen reader users and in situations where images cannot be displayed.

Instead of this:

<img src="logo.png">
<img src="banner.jpg" alt="banner">
<img src="icon.svg" alt="image">

Do this:

<img src="logo.png" alt="Company Name Logo">
<img src="banner.jpg" alt="Team collaborating in modern office space">
<img src="icon.svg" alt="Download PDF" role="img">

For decorative images that don't add content value, use an empty alt attribute: 

<img src="decorative-line.png" alt="" role="presentation">

4. Color and Contrast

Ensure sufficient color contrast between text and its background. WCAG 2.1 requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text.

Poor contrast: difficult to read

Good contrast: easy to read

Never rely on color alone to convey information:

Instead of this:

<p>Fields in <span style="color: red;">red</span> are required.</p>
<div class="status-green">Success</div>

Do this:

<p>Fields marked with an asterisk (*) are required.</p>
<div class="status-success">
  <i class="fas fa-check-circle" aria-hidden="true"></i> Success
</div>

5. ARIA (Accessible Rich Internet Applications)

ARIA attributes can enhance accessibility when HTML semantics alone aren't enough. However, they should be used judiciously and correctly. 

<button 
  aria-expanded="false" 
  aria-controls="navigation-menu"
>
  <span class="sr-only">Menu</span>
  <i class="fas fa-bars" aria-hidden="true"></i>
</button>

<nav id="navigation-menu" aria-label="Main navigation" hidden>
  <!-- Navigation items -->
</nav>

The First Rule of ARIA

If you can use a native HTML element or attribute with the semantics and behavior you require, do so instead of using ARIA. Only use ARIA when you can't achieve the semantics with standard HTML.

6. Forms and Error Handling

Create accessible forms by using proper labels, describing inputs clearly, and providing helpful error messages. 

<form>
  <div class="form-group">
    <label for="email">Email address:</label>
    <input 
      type="email" 
      id="email" 
      name="email" 
      aria-required="true"
      aria-describedby="email-hint"
    >
    <p id="email-hint" class="form-hint">We'll never share your email with anyone else.</p>
  </div>
  
  <div class="form-group">
    <label for="password">Password:</label>
    <input 
      type="password" 
      id="password" 
      name="password"
      aria-required="true"
      aria-describedby="password-error"
      aria-invalid="true"
    >
    <p id="password-error" class="form-error">
      <i class="fas fa-exclamation-circle" aria-hidden="true"></i>
      Password must be at least 8 characters
    </p>
  </div>
  
  <button type="submit">Sign In</button>
</form>

7. Document Structure and Headings

Use headings (h1-h6) to create a logical document structure. Screen reader users rely on headings to navigate content efficiently. 

<h1>Main Page Title</h1>
<section>
  <h2>Section Title</h2>
  <p>Section content...</p>
  
  <article>
    <h3>Article Title</h3>
    <p>Article content...</p>
    
    <section>
      <h4>Subsection Title</h4>
      <p>Subsection content...</p>
    </section>
  </article>
</section>

Testing for Accessibility

Regular testing is essential to ensure your website remains accessible. Here are some methods you should incorporate into your development workflow:

Automated Testing

Use tools like Axe, WAVE, or Lighthouse to identify common accessibility issues automatically.

Keyboard Testing

Navigate your entire site using only the keyboard to ensure all functionality is accessible.

Screen Reader Testing

Test with screen readers like NVDA, JAWS, or VoiceOver to experience how users with visual impairments interact with your site.

User Testing

Involve people with disabilities in your testing process to get authentic feedback on real-world usage.

Integrating Accessibility into Your Workflow

Accessibility should be considered at every stage of the development process, not as an afterthought. Here's how to integrate it into your workflow:

  1. Planning: Include accessibility requirements in project specifications and user stories
  2. Design: Consider color contrast, text size, and layout from the beginning
  3. Development: Use semantic HTML, implement keyboard support, and follow ARIA patterns
  4. Testing: Include accessibility testing in your QA process
  5. Deployment: Include accessibility features in documentation
  6. Maintenance: Regularly audit your site for accessibility issues

Conclusion

Web accessibility isn't just about compliance or avoiding legal issues—it's about creating a better user experience for everyone. By implementing these best practices, you're not only making your site usable for people with disabilities but also improving it for all users across different devices and situations.

Remember that accessibility is a journey, not a destination. Start with small improvements and gradually build your knowledge and implementation. The most important step is to begin considering accessibility in your development process.

Ready to get started with accessibility?

Try running a quick scan of your website with the WAVE Web Accessibility Evaluation Tool to identify immediate improvement opportunities.
